# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv`. Прогнозировался временной ряд, обозначенный как `target`.
- Данные были корректно разбиты на `train / validation / test` по времени (70/15/15).
- Сравнивались 4 подхода: наивный прогноз (B1), скользящее среднее (B2), гребневая регрессия на лаговых и календарных признаках (B3) и рекуррентная нейросеть GRU (R1).
- Лучшим подходом по валидационной метрике MAE оказалась `GRU`, однако на тестовой выборке лучший результат показала `Ridge`.

## 2. Среда и воспроизводимость

- Python: 3.10
- pandas / numpy / scikit-learn: 1.5.3 / 1.24.3 / 1.3.0
- torch: 2.1.0
- Устройство: CPU
- Seed: 42
- Как запустить: открыть `HW12.ipynb` и выполнить Run All.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: ежедневная (Daily)
- Горизонт прогноза: 1 шаг вперед (next value)
- Размер датасета: 1000 строк
- Диапазон дат: с 2020-01-01 по 2022-09-27
- Есть ли дополнительные внешние признаки: Нет, использовался только сам ряд.
- Комментарий: Ряд демонстрирует умеренный восходящий тренд, ярко выраженную недельную сезонность и присутствуют случайные выбросы. Пропуски отсутствуют. Задача представляет собой классический пример прогнозирования для временных рядов с трендом и сезонностью.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Разбиение: 70% train, 15% validation, 15% test.
- Интервалы: train: 2020-01-01 — 2021-12-19; val: 2021-12-20 — 2022-04-20; test: 2022-04-21 — 2022-09-27.
- Почему `random split` здесь некорректен: При случайном перемешивании модель получила бы доступ к будущей информации во время обучения, что сделало бы оценку качества на тесте нереалистично оптимистичной и не соответствовало бы реальному сценарию использования.

### 4.2. Признаки для baseline-моделей

- Лаги: 1, 7, 14.
- Rolling-признаки: скользящее среднее и стандартное отклонение за окно 7 (с shift, чтобы не использовать текущее значение).
- Календарные признаки: день недели, месяц, их синус/косинус кодировка.
- Пропуски, возникшие после создания лагов и rolling, были удалены.
- Масштабирование: `StandardScaler` использовался только для признаков в модели `Ridge`, обучение масштаба происходило на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- B1 (`naive-last`): прогноз равен последнему известному значению ряда.
- B2 (`moving-average`): прогноз как среднее за последние 7 наблюдений.
- B3 (`ridge-lag-features`): линейная модель с L2-регуляризацией на признаках из прошлого.
- R1 (`gru-forecast`): рекуррентная сеть GRU с 2 слоями и 64 скрытыми нейронами, обучаемая на последовательностях длины 30.
- Основная метрика для выбора лучшей модели: MAE.
- Дополнительные метрики: RMSE, MAPE.
- `window_size` для `GRU`: 30
- Batch size: 64
- Максимальное число эпох: 100 (ранняя остановка при 10 эпохах без улучшения)
- Optimizer: Adam
- Learning rate: 0.001
- Лучшая модель сохранялась по наименьшему значению MSE на валидационной выборке.

## 7. Анализ

Наивный baseline (B1) показал неожиданно хороший результат (val_MAE = 6.445), что говорит о высокой инерционности ряда — последнее значение является довольно хорошим прогнозом на следующий день. Скользящее среднее (B2) оказалось значительно хуже (val_MAE = 12.702), вероятно, из-за того, что оно слишком сильно сглаживает ряд и не успевает реагировать на быстрые изменения.

Добавление лаговых признаков и скользящих статистик в `Ridge` (B3) дало результат, сопоставимый с наивным подходом по MAE (6.445), но лучше по RMSE и MAPE. Это показывает, что даже простая линейная модель на правильно подобранных признаках может быть эффективной.

`GRU` (R1) превзошла все остальные модели на валидации по всем трем метрикам. Это подтверждает способность нейросетей улавливать более сложные нелинейные зависимости в данных. Однако на тестовой выборке `GRU` показала результат хуже, чем `Ridge`. Это классический признак переобучения: модель слишком подстроилась под валидационные данные и хуже обобщается на новые.

Наибольшие ошибки модели, судя по визуализации, приходятся на моменты резких скачков ряда, что типично для прогнозирования волатильных временных рядов. В такой задаче утечки данных могли бы возникнуть при использовании `center=True` в скользящем окне или при неправильной нормализации (fit на всей выборке). Выбор лучшей модели по `validation`, а не по `test`, принципиален для объективной оценки, хотя в данном случае это привело к выбору модели, которая оказалась хуже на тесте.

## 8. Итоговый вывод

В качестве базового подхода для подобной задачи я бы рекомендовал использовать `Ridge` на лаговых и календарных признаках. Несмотря на то, что `GRU` показала лучшие результаты на валидации, `Ridge` оказалась более стабильной и показала лучший результат на тестовой выборке. Кроме того, линейная модель проще в интерпретации, быстрее обучается и менее склонна к переобучению.

Если требуется еще большая точность, переход к `GRU` может быть оправдан, но необходимо уделить особое внимание борьбе с переобучением: использовать больше регуляризации (dropout, weight decay), увеличить размер валидационной выборки или применить более сложные методы валидации (например, walk-forward validation).

Главный вывод по валидации временных рядов: случайное разбиение данных (random split) в корне неверно для прогнозирования. Использование temporal split — это не просто рекомендация, а обязательное требование для получения реалистичной оценки качества модели, пригодной для внедрения. Также важно помнить, что лучшая модель на валидации не всегда оказывается лучшей на тесте, особенно при наличии переобучения.
